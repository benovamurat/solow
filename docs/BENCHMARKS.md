# Solow — Performance benchmarks

Solow's numerical results match an authoritative reference to ~1e-8…1e-13 (see the
verification suite). This page tracks **speed**. There are two independent measurements,
both reproducible from the repo:

1. **Absolute throughput** of the native Rust estimators, measured by `criterion`
   (`solow-bench`). This is the honest "how fast is the Rust code" number, with
   statistical confidence intervals and no reference involved.
2. **Speed relative to an authoritative reference**, measured through the `solow-py`
   Python bindings on identical data, so the comparison is apples-to-apples (same
   inputs, same reported quantity).

All numbers below were measured on **one development machine** and should be read as
indicative, not as a spec. They will differ on other hardware.

### Machine & toolchain

| | |
| --- | --- |
| CPU | Apple M3 Pro (12 cores), 36 GB RAM |
| OS | macOS (Darwin 25.x) |
| Toolchain | `rustc 1.96.0`, release profile (`opt-level` per workspace) |
| Backend | pure-Rust numeric core, **single-threaded** (no system BLAS/LAPACK) |
| Harness | `criterion` (100 samples/bench, 3 s warm-up, 5 s collection) |

## 1. Native estimator throughput (criterion)

Reproduce with:

```bash
cargo bench -p solow-bench
```

All inputs are generated by a deterministic LCG (no clock seed), so the workloads are
reproducible. These benchmarks measure **speed only** — correctness is covered by the
fixture and validation suites. Times are criterion's reported median with the 95 %
confidence interval `[low, high]`.

| Benchmark | Workload | Median | 95 % CI |
| --- | --- | --- | --- |
| `ols_fit/n100_k5`     | OLS fit, n=100, 5 predictors + intercept   | **6.09 µs**  | [6.07, 6.10] µs |
| `ols_fit/n1000_k5`    | OLS fit, n=1000, 5 predictors + intercept  | **57.8 µs**  | [56.7, 59.6] µs |
| `glm_poisson_irls/n500_k5` | GLM Poisson IRLS (log link), n=500    | **178 µs**   | [151, 213] µs |
| `logit_newton/n500_k5`| Logit Newton fit, n=500                    | **277 µs**   | [269, 291] µs |
| `linalg/svd_200x50`   | Economy SVD of a 200×50 matrix             | **2.94 ms**  | [2.93, 2.94] ms |
| `linalg/eigh_100x100` | Symmetric eigendecomposition, 100×100      | **73.6 ms**  | [73.4, 73.9] ms |
| `dist_throughput/norm_cdf` | normal CDF, 1024 points              | **118 µs**   | [117, 119] µs |
| `dist_throughput/norm_ppf` | normal PPF (inverse CDF), 1024 points| **233 µs**   | [232, 235] µs |
| `dist_throughput/t_cdf_df5`| Student-t CDF (df=5), 1024 points    | **75.8 µs**  | [75.5, 76.3] µs |
| `dist_throughput/t_ppf_df5`| Student-t PPF (df=5), 1024 points    | **1.48 ms**  | [1.39, 1.60] ms |

Notes on spread:

* The GLM-Poisson and Logit benches show wider CIs because each timed iteration runs a
  full IRLS / Newton loop to convergence, so iteration count (and therefore time) varies
  slightly with the deterministic-but-data-dependent workload.
* `eigh_100x100` and `t_ppf_df5` are the two slowest single calls; criterion warned it
  could not fit 100 samples in the default 5 s window for both and extended the
  collection time automatically. The reported medians are still over 100 samples.
* The distribution benches time **1024 evaluations per iteration**, so per-call cost is
  roughly the table value divided by 1024 (e.g. `norm_cdf` ≈ 115 ns per evaluation).

## 2. Speed vs. an authoritative reference (through `solow-py`)

Reproduce with:

```bash
SOLOW_REFERENCE=<reference-import-name> python3 -W ignore \
    tools/reference/bench_vs_reference.py
```

This builds `solow-py` (release) and times the same fit in Solow and in the reference on
identical NumPy arrays. `ratio = solow_time / reference_time` — **< 1.0 means Solow is
faster.** The figures below are from two consecutive runs on the machine above; where the
two runs differ at large `n` (single-run wall-clock medians are noisy), a range is shown.

### Linear regression (OLS)

| size | Solow | reference | ratio | verdict |
| --- | --- | --- | --- | --- |
| n=1000, k=10   | 0.10 ms | 0.18 ms | **0.56–0.58×** | faster |
| n=10000, k=20  | 3.0 ms  | 2.6–3.0 ms | 1.04–1.14× | ~parity |
| n=100000, k=30 | 65–67 ms | 54–63 ms | 1.07–1.19× | ~parity, slightly slower |

### Generalized linear models (Poisson, IRLS)

| size | Solow | reference | ratio | verdict |
| --- | --- | --- | --- | --- |
| n=500, k=5    | 0.12 ms | 0.58 ms | **0.21×** | faster |
| n=5000, k=10  | 3.6 ms  | 4.2 ms  | **0.84–0.86×** | faster |
| n=50000, k=15 | 89–91 ms | 47–49 ms | 1.84–1.91× | slower |

### Discrete choice (Logit, Newton)

| size | Solow | reference | ratio | verdict |
| --- | --- | --- | --- | --- |
| n=1000, k=5   | 0.37–0.38 ms | 0.48 ms | **0.78–0.79×** | faster |

## Reading the results

The shape of the story is consistent across all three model families:

* **On small and medium problems — the common case — Solow is faster than the
  reference**, often substantially (GLM-Poisson at n=500 is ~5× faster; OLS at n=1000 is
  ~1.8× faster). At these sizes the reference pays fixed Python/dispatch overhead that the
  native Rust path avoids.

* **At very large `n` Solow is slower**, by ~1.1× (OLS) up to ~1.8–2.3× (large GLM-Poisson
  and Logit). The reference there is dominated by a **multithreaded** BLAS/LAPACK, while
  Solow's default path is **single-threaded pure Rust**. The crossover sits around
  n≈10⁴: below it Solow's lower overhead wins, above it the reference's parallel BLAS
  wins.

## What changed (M1) and what is still open

The original linear-algebra core used a one-sided Jacobi SVD on the hot path, which is
asymptotically slower than the reference's LAPACK backend (OLS at n=10000 was historically
~9× slower). M1 added a **column-major, vectorized Householder-QR fast path**
(`lstsq_qr`) used for full-rank designs in `LinearModel::fit` and inside the GLM IRLS loop
(the SVD pseudoinverse remains the rank-deficient fallback). This is the same
least-squares solution to within rounding, so every correctness test still passes, and it
turned the large-`n` OLS result from ~9× slower into roughly parity while making the
common small/medium case **faster** than the reference.

The remaining large-`n` gap (worst case: large Logit/GLM-Poisson at ~1.8–2.3×) is the
single-threaded-vs-parallel-BLAS gap. The plan closes it with:

- an optional `lapack` feature (LAPACK/BLAS backend) for users who want maximum speed,
- `rayon` parallelism of the QR reflector application and the IRLS weight construction,
- avoiding per-iteration design re-whitening in GLM/discrete.

The design goal is: **faster than the reference on small/medium problems (the common
case), and within ~1.2× at very large scale with the pure-Rust default — at parity or
better with the optional LAPACK backend.** OLS already meets the large-scale target
(~1.1×); the iterative GLM/discrete fits are the remaining work.
